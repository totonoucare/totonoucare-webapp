# v7.71 Daily Care実行記録

## 目的

予報ページのDaily Careカードを、ケア提案の表示だけでなく、ユーザーが実際に行った具体的ケアの記録入口として使う。

ユーザー体験は次の流れとする。

```text
前日: 「明日に向けた今夜のケア」で、行った項目を「やってみた」
当日: 「今日のケア」で、行った項目を「やってみた」
夜:   記録ページで実感○△×と、当日ケアがつらさの前か後かを確認
後日: 固定された予報条件を物差しに、具体的ケア・タイミング・実感をAIで振り返る
```

ケアの実行記録は予報ロジックを変更しない。予報はベース体質×天気ストレスの固定ロジックで算出し、記録は振り返りとAI分析にのみ使う。

## 画面仕様

### 予報ページ

- 暮らす、食べる、ほぐすの具体的な各項目に「やってみた」を表示する。
- 押すと「記録済み」になり、もう一度押すと取り消せる。
- 食べるの「控えたいこと」は、実行ではなく意識した行動なので「意識した」と表示する。
- 前日の明日カードで記録したものは、翌日の対象日に対する確実な先回りケアとして保存する。
- 当日の今日カードで記録したものは、症状との前後関係を未確定で保存する。
- ケアナビのキャラクターは、ルールベースで選ばれたケアを案内し、記録導線を説明する。AIが自由生成したケアであるとは表示しない。

### 記録ページ

具体的ケアが予報ページですでに記録されている場合、従来の「ケアはした？」「暮らす／食べる／ほぐす」を再質問しない。

- 実感○△×を記録する。
- 前夜ケアは先回りとして自動表示する。
- 当日ケアがある場合だけ、1日単位で次を確認する。
  - だいたい前にできた
  - つらくなってから
  - 前後どちらも
  - 覚えていない
- 予報ページで記録した具体的ケアを一覧表示し、今日の記録から予報ページへ戻って修正できる。
- 具体的ケアがない日は、従来の簡易的なケア分野記録を継続して利用できる。

## データ仕様

新規テーブル `public.radar_care_actions` を使用する。

主な意味:

- `target_date`: ケアで備える対象日
- `source_date`: ケアをチェックした日
- `source_mode = tomorrow`: 前日に表示された明日カード
- `source_mode = today`: 対象日当日に表示された今日カード
- `item_key`: 表示された具体的ケアの安定キー
- `item_snapshot`: チェック時点の表示内容と予報・ケアルール文脈
- `timing_relation = previous_night`: 確実な先回りケア
- `same_day_*`: 記録ページで本人が確認した当日ケアの前後関係

同じ対象日・表示モード・具体的ケアは1件に保つ。

## AI分析での扱い

AIへは以下を区別して渡す。

- `displayed_care`: アプリが提案したケア
- `performed_care_items`: 本人が「やってみた」と記録した具体的ケア

分析条件は、主な天気ストレス、予報モード、体調ゆらぎ度の10点帯を基本とする。

前夜と当日の両方にケアがある日は、タイミング区分が重複する。区分ごとの日数を単純合計しない。

同日に複数のケアを行った場合、その日の実感は各ケアと関連するが、単独ケアの効果とは断定しない。具体的ケアの集計は「似た条件で再び試す候補」として扱う。


## DB適用

アプリのデプロイ前に次のmigrationをSupabase SQL Editorで実行する。

```text
supabase/migrations/20260713_create_radar_care_actions_v771.sql
```

適用後、必要に応じて次の確認SQLを実行する。

```text
supabase/checks/20260713_check_radar_care_actions_v771.sql
```

DBを適用していない環境では、予報とケアの閲覧は続けられるが、「やってみた」記録は準備中として無効になる。

## ロールバック

```text
supabase/migrations/20260713_rollback_radar_care_actions_v771.sql
```

ロールバックは具体的ケア記録を削除する。`mixed`になっている旧集約記録は`unknown`へ戻してから旧制約を復元する。
