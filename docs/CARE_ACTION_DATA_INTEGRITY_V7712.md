# v7.71.2 ケア記録データ整合性修正

## 目的

v7.71で追加したDaily Careの具体的ケア記録を、有料AI分析の長期データとして安全に使えるようにする。

## 修正内容

### 1. 具体的ケアIDを表示順から分離

旧キーは表示順のindexを含んでいたため、並び替えで同じケアが別物になり得た。

v7.71.2では、次の意味上の情報からv2キーを作る。

- ツボ: ツボコード。なければツボ名
- 食べる: カード種別と食品名
- 暮らす: ルールID。なければ表示内容

旧v7.71行もitem_snapshotからv2のcanonical_keyへ正規化する。DB行を一括書き換えず、読み込み・解除・集計時に互換処理する。

### 2. 具体的ケアとまとめ入力を分離

`radar_reviews.prevent_level`は既存画面互換の集約値として残す。

新規列は、ユーザーが従来のまとめ質問で明示したケアだけを保持する。

- `manual_prevent_level`
- `manual_care_domains`
- `manual_care_timing`

集約値は次のように同期する。

```text
具体的ケアあり → prevent_levelは最低1
具体的ケアなし → manual_prevent_levelをそのまま使用
care_domains / care_timing → 具体的ケアとまとめ入力を統合
```

具体的ケアをすべて解除した場合も、まとめ入力が0なら集約値を0へ戻す。

### 3. 記録ページから直近7日を修正

記録ページで次を行える。

- 保存済みの具体的ケアを個別削除
- 提案とは別に行ったケアを追加
- 追加時に暮らす／食べる／ほぐすと、おおまかなタイミングを指定

前日に表示された明日カードのケアも、対象日の記録ページから削除できる。

### 4. 同じケアを前夜と当日に行った場合

同じケアの実施日数は1日として数える。

ただしタイミングは失わず、次をそれぞれ保持する。

- 前夜
- 当日のつらさの前
- 当日のつらさの後
- 前後両方／不明

タイミング別日数は重複し得るため、AIプロンプトでは単純合計しないよう指定している。

### 5. AIへのデータ最小化

`checked_at`はボタンを押した時刻であり、実際のケア実施時刻とは限らない。

- DB: 監査・更新順のため保持
- AI送信: 除外
- AI分析: `source_mode`と`timing_relation`を使用

### 6. 「控えたいこと」の詳細表示

具体的ケア一覧で`detail`も表示する。ラベルが「控えたいこと」だけでも、何を控えたのか確認できる。

## 必須migration

v7.71のmigration適用後に、次を実行する。

```text
supabase/migrations/20260713_add_care_data_integrity_v7712.sql
```

確認:

```text
supabase/checks/20260713_check_care_data_integrity_v7712.sql
```

このmigrationは`radar_care_actions`を削除しない。
